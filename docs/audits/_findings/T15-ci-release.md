- **Quote**:

Pin Node version via `actions/setup-node@<SHA>` with `node-version-file: '.nvmrc'`. Set `permissions: contents: read` at workflow level. ### [MEDIUM | confidence=high] CHANGELOG.md does not conform to Keep a Changelog format

- **File**: `CHANGELOG.md:1-22`
- **Quote**:

  ```markdown
  # changelog

  all notable changes to this project. dates are ISO format (YYYY-MM-DD).

  ## [6.0.0] - 2026-04-06

  ### added

  - **beginner operations toolkit**: added `codex-help`, `codex-setup` (with `wizard` mode + fallback), `codex-doctor` (`fix` mode), and `codex-next` for guided onboarding and recovery.
  ```

- **Issue**: [Keep a Changelog v1.1.0](https://keepachangelog.com/en/1.1.0/) specifies: (a) title `# Changelog` (capital C), (b) preamble linking the spec and stating semver compliance, (c) an `## [Unreleased]` section at the top for upcoming changes, (d) category headings with **capitalized** first letter (`### Added`, `### Changed`, `### Deprecated`, `### Removed`, `### Fixed`, `### Security`). The current CHANGELOG violates all four: (a) `# changelog` lowercase; (b) preamble is `all notable changes to this project. dates are ISO format` with no spec link and no semver statement; (c) no `## [Unreleased]` section, so in-progress changes have no canonical landing zone; (d) headings are lowercase and include non-standard categories like `### breaking` (`CHANGELOG.md:171`, v5.0.0), `### metadata` (`CHANGELOG.md:287`, v4.12.4), `### migration` (`CHANGELOG.md:509`, v4.9.0), `### config` (`CHANGELOG.md:397`, v4.10.0). The `breaking` section is especially important — Keep a Changelog does not define it, and `### Changed` with a "**BREAKING**" prefix or explicit semver major-bump notice is the canonical approach. This drift makes automated changelog tooling (e.g. [release-please](https://github.com/googleapis/release-please), [changesets](https://github.com/changesets/changesets)) unable to parse the file, and weakens the changelog's consumer contract.
- **Recommendation**: Either (a) adopt Keep a Changelog strictly — capitalize title and section names, add `## [Unreleased]`, map `### breaking` → `### Changed` with "**BREAKING:**" prefix, map `### metadata`/`### migration`/`### config` into `### Changed` or remove — and add a preamble line `The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/), and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).`; or (b) declare an explicit internal format in `CONTRIBUTING.md` and link to it from the CHANGELOG preamble. Option (a) is recommended because it enables future adoption of changeset-style tooling and gives downstream consumers a stable machine-parseable contract.
- **Evidence**: `CHANGELOG.md:1` is literally `# changelog` (lowercase). `CHANGELOG.md:7` is `### added` (lowercase). Non-standard sections present: `CHANGELOG.md:171` `### breaking`, `CHANGELOG.md:287` `### metadata`, `CHANGELOG.md:397` `### config`, `CHANGELOG.md:509` `### migration`. Keep a Changelog spec: https://keepachangelog.com/en/1.1.0/. Direct read.
